# Выборочная маршрутизация Shadowrocket

Проект отправляет через выбранный в Shadowrocket прокси только:

- весь актуальный список `Russia/inside-clashx.lst` проекта itdoginfo;
- Telegram, WhatsApp и Discord (домены Telegram/Discord обновляются из отдельных списков itdoginfo, IP Telegram и домены WhatsApp хранятся локально);
- будущие ресурсы, добавленные в `rules/proxy-domains.list` или `rules/proxy-ips.list`.

Любой другой трафик совпадает с последним правилом `FINAL,DIRECT` и идёт напрямую через интернет-провайдера.

Apple в основном обрабатывается ранним правилом `rules/direct-apple.list`: основные домены Apple/iCloud и принадлежащая Apple сеть `17.0.0.0/8` получают `DIRECT` до общих proxy-списков. Выше него расположен небольшой `rules/proxy-apple-services.list` для тех узлов App Store и Apple Music, прямое подключение к которым стабильно блокируется или теряется на маршруте провайдера.

## Что означают DIRECT и обход Shadowrocket

`DIRECT` означает «установить соединение без прокси-сервера». Пакет всё ещё может пройти через TUN-интерфейс Shadowrocket, чтобы приложение смогло применить правила и DNS-логику, но он не отправляется на VPN/прокси-узел. В этом проекте нет HTTPS-дешифрования, rewrite, скриптов или блокировки, поэтому содержимое прямого трафика не изменяется.

Полностью не обрабатывать трафик можно только более ранним обходом:

- `bypass-system = true` оставляет поддерживаемые системные сетевые вызовы iOS вне прокси и снижает риск задержки push-уведомлений;
- `tun-excluded-routes` выводит из TUN локальные, link-local и multicast IP-диапазоны;
- `skip-proxy` используется для локальной сети и `captive.apple.com`.

Не следует добавлять Apple в `tun-excluded-routes`: адреса CDN динамические, а исключение маршрута действует по IP, до обычных правил Shadowrocket. Проект использует ранний `DIRECT`, системный DNS и `always-real-ip` для Apple, но точечно отправляет недоступные CDN-узлы App Store/Music через прокси. MITM для Apple не используется.

## Подготовка перед публикацией

Проект уже настроен для публичного репозитория `0xAlexFox/routing_conf`, ветка `master`. Для повторной генерации итогового конфига выполните:

```sh
./scripts/configure.sh
```

Если репозиторий когда-нибудь изменится, передайте новые значения явно:

```sh
./scripts/configure.sh GITHUB_LOGIN REPOSITORY BRANCH
```

Скрипт создаст `routing.conf`, подставит правильные raw-URL и выполнит проверку. После этого загрузите весь каталог в публичный GitHub-репозиторий. Для приватного репозитория обычные raw-URL без авторизации в Shadowrocket не подойдут.

Проверьте проект вручную в любое время:

```sh
./scripts/validate.sh
```

## Установка в Shadowrocket

1. Добавьте ваш прокси/VPN-узел в Shadowrocket. Для звонков Telegram, WhatsApp и Discord узел должен поддерживать UDP.
2. Откройте `Конфигурация` → `+` → `Скачать файл` и вставьте:

   ```text
   https://raw.githubusercontent.com/0xAlexFox/routing_conf/refs/heads/master/routing.conf
   ```

3. Выберите скачанный `routing.conf` → `Использовать конфигурацию`.
4. На главном экране установите `Глобальная маршрутизация` в режим `Конфигурация`.
5. Выберите нужный прокси-узел и включите Shadowrocket.

Конфиг использует встроенную политику `PROXY`, то есть текущий выбранный в приложении узел. Серверные данные и секреты в репозиторий добавлять не нужно.

## Добавление нового ресурса

Для домена добавьте строку в `rules/proxy-domains.list`:

```text
DOMAIN-SUFFIX,example.com
```

Для отдельной сети добавьте строку в `rules/proxy-ips.list`:

```text
IP-CIDR,203.0.113.0/24
```

После commit/push в Shadowrocket нажмите на конфигурацию и выберите обновление. Удалённые RULE-SET также обновляются при повторном применении конфигурации.

## DNS, IPv6 и UDP

- Прямые соединения используют системный DNS (`dns-direct-system = true`). Доменные правила прокси имеют `force-remote-dns`, чтобы заблокированный или подменённый локальный DNS не мешал соединению.
- IPv6 отключён, чтобы трафик не обходил IPv4-only прокси. Включайте его только если прокси-узел и сеть гарантированно поддерживают IPv6, затем проверьте утечки.
- `udp-policy-not-supported-behaviour = REJECT` не позволяет звонкам незаметно уйти напрямую при отсутствии UDP у прокси. В таком случае нужно сменить узел, а не переключать параметр на `DIRECT`.
